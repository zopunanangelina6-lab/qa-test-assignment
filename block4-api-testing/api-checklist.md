 Чек-лист тестирования контроллера /api/v2/orders

 1. GET /api/v2/orders (Получение списка заказов)

 Успешные сценарии:
- 200 OK — возвращён список заказов
- Response содержит пагинацию (`page`, `per_page`, `total_count`)
-  Content-Type: `application/json`
-  Заказы отсортированы по дате создания (новые сверху) по умолчанию

 Ошибки и статус-коды:
- 401 Unauthorized — отсутствует или невалидный токен авторизации
- 403 Forbidden — нет прав на просмотр списка (например, роль "Гость")
- 500 Internal Server Error — падение БД или внутренняя ошибка сервера
- 503 Service Unavailable — сервис временно недоступен


 2. GET /api/v2/orders/{id} (Получение конкретного заказа)

 Успешные сценарии:
 - 200 OK — возвращён полный объект заказа со всеми полями

Ошибки и статус-коды:
- 401 Unauthorized — нет токена
- 403 Forbidden — попытка получить чужой заказ (нет прав доступа)
- 404 Not Found — заказ с таким ID не существует
- 400 Bad Request — невалидный формат ID (например, строка вместо числа)
- 500 Internal Server Error — падение БД


3. POST /api/v2/orders (Создание заказа)

 Успешные сценарии:
- 201 Created — заказ успешно создан
-  Header `Location` содержит URL созданного ресурса (`/api/v2/orders/{id}`)
-  Response body содержит созданный заказ
-  Заказ корректно сохранён в БД

 Ошибки валидации и статус-коды:
- 400 Bad Request — некорректный JSON (синтаксическая ошибка, missing fields)
-  422 Unprocessable Entity — бизнес-валидация не пройдена:
-  Вес груза > 20000 кг
-  Объем > 82 м³
  -  Дата погрузки в прошлом
  -  Город отправления = город назначения
-  409 Conflict — дублирование `idempotency-key` (заказ уже существует)
-  401 Unauthorized — нет токена авторизации
-  500 Internal Server Error — падение БД при сохранении

 4. PATCH /api/v2/orders/{id} (Обновление заказа)

 Успешные сценарии:
-  200 OK — заказ обновлён
-  Response body содержит обновлённый заказ

 Ошибки и статус-коды:
-  401 Unauthorized — нет токена
-  403 Forbidden — нет прав на редактирование чужого заказа
-  404 Not Found — заказ не существует
-  409 Conflict — конфликт версий (optimistic locking, если заказ уже изменён другим пользователем)
-  422 Unprocessable Entity — нельзя изменить заказ в финальном статусе (например, "Доставлен")
-  500 Internal Server Error — падение БД


5. DELETE /api/v2/orders/{id} (Удаление/Отмена заказа)

Успешные сценарии:
-  204 No Content — заказ успешно удалён/отменён (тело ответа пустое)

 Ошибки и статус-коды:
-  401 Unauthorized — нет токена
-  403 Forbidden — нет прав на удаление
-  404 Not Found — заказ не существует
-  409 Conflict — заказ уже в работе/доставлен (нельзя удалить)
-  500 Internal Server Error — падение БД

6. Интеграционные и нефункциональные проверки

Идемпотентность и повторные запросы:
-  Повторный `POST` с тем же `idempotency-key` возвращает `409 Conflict` или `200 OK` с существующим заказом (не создаёт дубль)
-  Повторный `DELETE` того же ID возвращает `404 Not Found` или `204 No Content` (не падает с ошибкой)

 Rate Limiting (Ограничение частоты запросов):
-  Превышение лимита запросов (например, >100 req/min) → **429 Too Many Requests**
-  В ответе присутствуют headers: `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`

 Пагинация, фильтрация и сортировка (для GET /api/v2/orders):
-  Параметры `limit` и `offset` (или `page` и `per_page`) работают корректно
-  Фильтрация по статусу (`?status=active`) возвращает только нужные заказы
-  Сортировка (`?sort=created_at:desc`) меняет порядок выдачи
-  Невалидные параметры сортировки возвращают **400 Bad Request**

 Безопасность:
-  HTTPS обязателен (HTTP редиректит на HTTPS или возвращает ошибку)
-  Токены авторизации не передаются в URL (только в `Authorization: Bearer <token>`)
-  SQL-инъекции в параметрах фильтрации экранированы
-  XSS в текстовых полях заказа (название, комментарий) предотвращён (данные санитизируются)

 Производительность:
-  Response time < 500ms для 95% запросов (p95)
-  Response time < 1s для 99% запросов (p99)
-  Отсутствие N+1 запросов к БД при получении списка заказов
