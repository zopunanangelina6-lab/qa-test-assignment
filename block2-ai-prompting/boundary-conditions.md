 Граничные условия для тестирования формы "УЛ Лайт"

 1. Вес груза (максимум 20 000 кг)

| Значение | Ожидаемое поведение | Критичность | Почему важно |
|----------|---------------------|-------------|--------------|
| 0 | Ошибка валидации | Critical | Нулевой вес = фиктивная заявка, попытка обхода биллинга |
| 0.1 | Принято | Medium | Минимальный вес (например, документы) |
| 19999.99 | Принято | Low | Почти граница, проверка точности |
| 20000 | Принято (граница) | High | Максимально допустимый вес по закону |
| 20000.01 | Ошибка валидации | High | Превышение лимита = штраф для перевозчика |
| -1 | Ошибка валидации | High | Отрицательный вес физически невозможен |
| "abc" | Ошибка типа данных | Medium | Неверный тип, проверка фронтенд-валидации |
| 25000 (через Postman) | 422 Unprocessable Entity | Critical | Обход фронтенда — бэкенд обязан валидировать |


 2. Объем груза (максимум 82 м³)

| Значение | Ожидаемое поведение | Критичность | Почему важно |
|----------|---------------------|-------------|--------------|
| 0 | Ошибка валидации | High | Нулевой объем невозможен |
| 0.1 | Принято | Medium | Минимальный объем |
| 81.99 | Принято | Low | Почти граница |
| 82 | Принято (граница) | High | Максимум для стандартной фуры (Еврофура) |
| 82.01 | Ошибка валидации | High | Превышение вместимости ТС |

Дополнительная проверка:
- Объемный вес: Если вес = 100 кг, а объем = 80 м³, система должна рассчитать объемный вес (обычно 1 м³ = 167 кг) и использовать большее значение для тарификации.


 3. Дата погрузки

| Сценарий | Ожидаемое поведение | Критичность | Почему важно |
|----------|---------------------|-------------|--------------|
| Сегодня | Принято | High | Можно заказать на сегодня (если успевает логистика) |
| Вчера | Недоступно в датапикере | Critical | Нельзя выбрать прошедшую дату (нарушение логики) |
| +365 дней | Принято | Medium | Планирование на год вперед |
| 29 февраля (високосный год)| Принято | Medium | Проверка корректности календаря |
| Пустое значение | Ошибка валидации | High | Обязательное поле |
| 31.02.2026 (через API) | 422 Unprocessable Entity | High | Несуществующая дата — бэкенд должен отклонить |


 4. Города (автокомплит)

| Сценарий | Ожидаемое поведение | Критичность | Почему важно |
|----------|---------------------|-------------|--------------|
| Отправление = Назначение | Ошибка "Маршрут не найден" | Critical| Бессмысленная перевозка, трата ресурсов |
| Несуществующий город | Ошибка валидации | High | Проверка по БД |
| Город без доставки (например, закрытая зона) | Сообщение о недоступности | High | Ограничения логистики (Крым, закрытые территории) |
| Очень длинное название | Корректное отображение, не ломает UI | Medium | UX (например, "Санкт-Петербург (аэропорт Пулково)") |
| Спецсимволы ("Москва-Сити") | Принято, если есть в БД | Low | Проверка экранирования |


 5. API "Рассчитать стоимость"

| Сценарий | Ожидаемое поведение | Критичность | Почему важно |
|----------|---------------------|-------------|--------------|
| Таймаут ответа (>10 сек) | Спиннер + сообщение "Повторите попытку" | High | Плохой UX, пользователь не должен ждать вечно |
| *Ошибка 500 | Тост "Произошла ошибка, попробуйте позже" | Critical | Фронтенд не должен падать с белым экраном |
| Невалидный JSON в ответе | Обработка ошибки, понятное сообщение | High | Проверка парсинга ответа |
| Двойной клик по кнопке | Отправляется только 1 запрос (идемпотентность) | Critical | Избежание дублирования расчетов/списаний |
| Медленный интернет (3G) | Спиннер, не блокирует UI | Medium | Проверка UX при плохой связи |


 Почему эти кейсы критичны?

 1. Финансовые риски
- Вес = 0 или отрицательный— попытка создать фиктивную заявку, обход биллинга
- Превышение 20000 кг / 82 м³— юридические ограничения перевозки, штрафы для перевозчика
- Двойной клик — если бы была оплата, это привело бы к двойному списанию

 2. Юридические ограничения
- Прошедшая дата — нарушение логистики, невозможно выполнить
- Закрытые территории — законодательные ограничения (например, Крым, закрытые города)

 3. UX и репутация
- Таймаут API без обработки — пользователь думает, что сайт завис, уходит к конкурентам
- Одинаковые города — бессмысленная заявка, трата времени пользователя и ресурсов системы

 4. Безопасность
- Обход фронтенд-валидации через Postman — бэкенд обязан валидировать данные, иначе злоумышленник может отправить вес 100000 кг и сломать логику тарификации

5. Специфика логистики
- Объемный вес— в логистике важен не только физический вес, но и объем (легкий, но огромный груз занимает много места в машине)
- Тип ТС — для разных весов/объемов нужны разные машины (Газель vs Фура), API должен возвращать подходящий тип


Что я бы добавила в реальном проекте

1. A/B тестирование формы:
   - Сравнить конверсию с валидацией на фронтенде vs только на бэкенде
   - Проверить, как пользователи реагируют на разные сообщения об ошибках

2. Мониторинг в продакшене:
   - Отслеживать частоту ошибок валидации (если много пользователей не могут заполнить форму — проблема UX)
   - Мониторить время ответа API (если >2 сек — деградация производительности)

3. Локализация:
   - Проверить, как форма работает на разных языках (если сервис международный)
   - Формат чисел (1000.5 vs 1,000.5)
